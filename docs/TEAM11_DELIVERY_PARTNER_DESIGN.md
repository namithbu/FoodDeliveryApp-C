# Team 11: Delivery Partner Management - Design Documentation

## Overview
The Delivery Partner Management module is responsible for managing delivery partners (drivers) in the FoodDeliveryApp. It provides functionality to register partners, manage their availability status, and retrieve available partners for order dispatch assignments.

**Team:** Team 11  
**Module:** edu.classproject.delivery  
**Dependencies:** None (leaf node in dependency graph)  
**Dependent Modules:** Dispatch Assignment (edu.classproject.dispatch)

---

## Module Scope

### Responsibilities
1. **Partner Registration**: Register new delivery partners with unique IDs and names
2. **Availability Management**: Toggle delivery partner availability (online/offline)
3. **Partner Discovery**: Retrieve list of currently available partners for dispatch assignment

### Out of Scope
- Authentication and authorization
- Payment/compensation for delivery partners
- Rating/performance tracking
- Location tracking (handled by tracking module)
- Communication/notifications (handled by notification module)

## Design Architecture

### 1. Core Components

#### 1.1 DeliveryPartner (Entity)
```
Data Model: Record-based immutable entity
Fields:
  - partnerId: String (unique identifier)
  - name: String (partner's name)
  - available: boolean (current availability status)
```

**Design Decision**: Using Java `record` provides:
- Immutability (IMMUTABLE VALUE OBJECT pattern)
- Automatic equals(), hashCode(), toString()
- Data integrity and thread-safety
- Reduced boilerplate code

#### 1.2 DeliveryPartnerService (Interface)
```
Public API for the module
Methods:
  - register(String name): DeliveryPartner
  - setAvailability(String partnerId, boolean available): void
  - getAvailablePartners(): List<DeliveryPartner>
```

**Design Decision**: Interface-based contract allows:
- Dependency inversion (SOLID: Dependency Inversion Principle)
- Multiple implementations (in-memory, database, mock)
- Easy testing and mocking
- Clear separation of concerns

## Design Patterns & Principles Summary

| Pattern/Principle | Application | Benefit |
|---|---|---|
| **Immutable Value Object** | DeliveryPartner record | Thread-safety, no side effects |
| **Dependency Injection** | Constructor or factory injection | Testability, loose coupling |

## State Management & Thread Safety

### Availability State Transitions
```
        ┌─────────┐
        │ Offline │
        └────┬────┘
             │
         register()
             │
        ┌────▼────┐
        │ Available│
        └────┬────┘
             │
    ┌────────┴────────┐
    │                 │
setAvailability     setAvailability
 (false)           (true)
    │                 │
┌───▼────┐       ┌────┴───┐
│ Offline │←──────│ Online │
└────────┘       └────────┘
```

 
**Recommended Implementation**:
```java
public List<DeliveryPartner> getAvailablePartners() {
    return Collections.unmodifiableList(
        partners.values().stream()
            .filter(DeliveryPartner::available)
            .collect(Collectors.toList())
    );
}
```

## Assumptions & Constraints

### Functional Assumptions
1. **Partner IDs are globally unique** - Generated by IdGenerator or database
2. **Availability is primary scheduling factor** - No location, capacity, or skill filtering
3. **No concurrent update conflicts** - Registration and availability changes are atomic
4. **Partner records never deleted** - Only availability toggled

### Non-Functional Assumptions
1. **Bounded partner set** - In-memory storage suitable for 1000-10,000 partners
2. **Read-heavy workload** - `getAvailablePartners()` called more than availability updates
3. **No persistence requirement** (in-memory implementation) - Data lost on restart
4. **Single-threaded or thread-safe runtime** - ConcurrentHashMap for safe access

### Future Considerations
1. **Scalability**: Move from in-memory to distributed cache (Redis) at 50K+ partners
2. **Persistence**: Add database repository for data durability
3. **Filtering**: Add location, vehicle type, skill-based filtering
4. **Rating/Performance**: Track delivery success rate and rating for dispatch priority
5. **Bulk Operations**: Add batch register/update for partner onboarding

## Acceptance Criteria Alignment

### ✅ Register partner and update availability supported
- `register(String name)`: Creates and registers new partner
- `setAvailability(String partnerId, boolean available)`: Updates availability

### ✅ getAvailablePartners only returns available partners
- Filters by `available=true` condition
- Returns empty list when no partners available
- Never returns null

### ✅ Tests include repeated toggles and unknown partner behavior
- Repeated toggles: Set available true→false→true
- Unknown partner: setAvailability on non-existent ID (should handle gracefully)
- Edge cases: Empty partner list, single partner, all partners unavailable

## API Documentation

### DeliveryPartnerService Interface

#### register(String name): DeliveryPartner
**Purpose**: Register a new delivery partner  
**Parameters**: 
- `name` (String): Partner's name, non-null, 1-50 characters

**Returns**: DeliveryPartner with unique ID, name, and available=true

**Throws**: 
- `IllegalArgumentException` if name is null/empty
- `RuntimeException` if ID generation fails

**Example**:
```java
DeliveryPartner partner = service.register("John Doe");
// Returns: DeliveryPartner("DP_UUID", "John Doe", true)
```

---

#### setAvailability(String partnerId, boolean available): void
**Purpose**: Update delivery partner's availability status  
**Parameters**:
- `partnerId` (String): Partner's unique ID
- `available` (boolean): New availability status (true=online, false=offline)

**Returns**: void

**Throws**: 
- `IllegalArgumentException` if partnerId is null/empty
- `NoSuchElementException` if partner not found (optional)

**Example**:
```java
service.setAvailability("DP_UUID", false); // Partner goes offline
service.setAvailability("DP_UUID", true);  // Partner goes online
```

---

#### getAvailablePartners(): List<DeliveryPartner>
**Purpose**: Retrieve list of currently available delivery partners  
**Parameters**: None

**Returns**: Unmodifiable list of available partners, or empty list if none available

**Throws**: No exceptions

**Performance**: O(n) where n = total partners; can be optimized with cached available set

**Example**:
```java
List<DeliveryPartner> available = service.getAvailablePartners();
// Returns: [DeliveryPartner("DP_1", "Alice", true), DeliveryPartner("DP_2", "Bob", true)]
```
